## 1. Introduction

### Testing Goal
Метою тестування є підтвердження того, що система `Web Time Tracker` коректно реалізує бізнес-процеси обліку робочого часу працівників, забезпечує достовірність відміток `Punch In / Punch Out`, захищене зберігання облікових даних, коректну обробку запитів на коригування часу, формування адміністративної аналітики та експорт звітів у форматі CSV. Додатково тестування має виявити дефекти, що впливають на функціональність, продуктивність, безпеку, сумісність та цілісність даних.

### Scope
У межах цього плану покривається серверна частина системи, реалізована на `Node.js` з використанням `Express 5.2.1`, драйвера `mongodb 7.1.1` та `bcryptjs 3.0.3`. Обсяг включає HTTP API, модулі запуску застосунку (`server.js`), доступу до БД (`db.js`), бізнес-логіку та маршрути (`app.js`), а також інтеграцію з базою даних `WebTimeTrackerDB` і колекціями `companies`, `users`, `time_logs`, `time_corrections`. Оскільки у поточному репозиторії відсутній окремий frontend-код, тестування графічних інтерфейсів розглядається як перевірка API-контрактів, що обслуговують вебкабінет, кіоск-режим та адміністративну панель, описані в попередніх лабораторних роботах.

## 2. Test Items

Тестуванню підлягають такі елементи системи:

- HTTP-сервер запуску застосунку та ініціалізація підключення до БД.
- Модуль підключення до `MongoDB` з роботою з базою `WebTimeTrackerDB`.
- REST API авторизації: `POST /api/auth/login`, `POST /api/auth/kiosk-login`.
- REST API управління компаніями та користувачами: `GET /api/companies`, `GET /api/companies/:companyId/users`, `POST /api/companies/:companyId/users`, `PATCH /api/users/:userId/reset-credentials`.
- REST API табелювання робочого часу: `GET /api/time-logs`, `POST /api/time-logs/punch-in`, `POST /api/time-logs/punch-out`, `PATCH /api/time-logs/:logId`.
- REST API коригувань часу: `POST /api/time-corrections`, `GET /api/time-corrections`, `PATCH /api/time-corrections/:requestId/review`.
- REST API адміністративної аналітики та звітності: `GET /api/admin/:companyId/dashboard`, `GET /api/reports/:companyId/time-logs.csv`.
- Колекції `companies`, `users`, `time_logs`, `time_corrections`, а також зв’язки через поля `company_id`, `user_id`, `time_log_id`.
- Механізми нормалізації та санітизації даних: приховування `password_hash`, `pin_hash`, `admin_password_hash`, конвертація `ObjectId` у рядкові ідентифікатори, нормалізація часових полів.
- Інтеграція з зовнішнім клієнтським контуром, що використовує API як backend для вебпорталу працівника, кіоск-режиму та адміністративної панелі.

## 3. Features to be Tested

Будуть перевірятися такі функціональні та нефункціональні характеристики:

- Авторизація працівника за `email/password` з підтримкою `bcrypt` і legacy-формату `sha256`.
- Авторизація у режимі кіоску за `organization_code` та персональним `PIN`.
- Створення нового користувача компанії з хешуванням пароля і PIN-коду.
- Скидання облікових даних користувача через оновлення пароля та/або PIN.
- Отримання списку компаній і користувачів компанії, включно з фільтрацією неактивних працівників.
- Реєстрація початку зміни (`punch-in`) з використанням серверного часу, збереженням `punch_method`, `note`, `status=active`, `duration_minutes=0`.
- Заборона створення більше однієї активної зміни для одного працівника.
- Реєстрація завершення зміни (`punch-out`) з перерахунком `duration_minutes`, зміною статусу на `approved` і конкатенацією приміток.
- Ручне редагування тайм-логів з перевіркою формату timestamp, статусів `active/approved/rejected`, логіки `punch_out > punch_in`, автоматичної зміни статусу та оновлення дати.
- Перегляд тайм-логів із фільтрами `company_id`, `user_id`, `status`, `from`, `to`.
- Створення запиту на коригування часу з автоматичним зв’язуванням до активного логу або до єдиного логу за відповідну дату.
- Обробка конфліктів при неоднозначному визначенні логу для коригування (`409`) та відсутності відповідного логу (`404`).
- Перегляд списку запитів на коригування з фільтрацією за компанією, користувачем і статусом.
- Розгляд коригування адміністратором зі статусами `approved/rejected`, оновленням пов’язаного тайм-логу у разі схвалення та фіксацією `reviewed_by` і `admin_comment`.
- Формування адміністративного дашборду компанії: кількість активних працівників, кількість pending-коригувань, останні відмітки та поточний статус кожного працівника.
- Генерація CSV-звіту з агрегуванням відпрацьованих хвилин/годин по працівниках і виключенням логів зі статусом `rejected`.
- Перевірка захисту конфіденційних полів: API-відповіді не повинні містити хеші паролів, PIN-кодів або пароль адміністратора компанії.
- Перевірка продуктивності для основних сценаріїв, зокрема часу відповіді для `login`, `punch-in`, `punch-out`, `dashboard`, `csv export`.
- Перевірка сумісності клієнтського доступу через сучасні браузери для сценаріїв використання вебінтерфейсу, описаних у SRS.

## 4. Features Not to be Tested

На цьому етапі не входять до обсягу тестування:

- Візуальне оформлення, адаптивна верстка та UX-деталі окремого frontend-застосунку, оскільки такий код відсутній у поточному репозиторії.
- Інтеграція з бухгалтерськими системами, ERP, payroll-сервісами або імпорт CSV до стороннього ПЗ.
- Налаштування `Nginx`, TLS-термінація, сертифікати HTTPS та балансування навантаження, бо ці компоненти не реалізовані в кодовій базі.
- Внутрішня коректність бібліотек `express`, `mongodb`, `bcryptjs`; перевіряється лише правильність їх інтеграції.
- Тестування відмовостійкості кластера MongoDB, реплікації, шардінгу та disaster recovery.
- Нативні мобільні застосунки, десктопні клієнти та офлайн-режим.
- Міграції історичних даних із паперових журналів або електронних таблиць.

## 5. Test Approach

### Testing Levels
- **Unit testing:** перевірка допоміжних функцій валідації, нормалізації даних, розрахунку тривалості зміни, обробки статусів та ідентифікаторів.
- **Integration testing:** перевірка взаємодії API-маршрутів із MongoDB, коректності CRUD-операцій і цілісності зв’язків між колекціями.
- **System testing:** наскрізна перевірка повних бізнес-сценаріїв працівника, кіоску та адміністратора через HTTP API.
- **Acceptance testing:** перевірка відповідності реалізації бізнес-вимогам із лабораторних робіт та SRS за участю замовника або представника предметної області.

### Methods
- **Manual testing:** exploratory testing, перевірка API в `Postman`/`Bruno`, перевірка CSV-вивантажень, валідація негативних сценаріїв.
- **Automated testing:** API-регресія за допомогою `Supertest` + `Jest` або `node:test`, колекції `Postman`/`Newman`, навантажувальні сценарії через `k6`.

### Test Types
- **Functional testing:** позитивні, негативні, граничні та альтернативні сценарії по всіх API.
- **Non-functional testing:** продуктивність, базова безпека, сумісність, надійність, цілісність даних, зручність експорту та стабільність при конкурентних операціях.

## 6. Pass/Fail Criteria

Фаза тестування вважається успішною, якщо:

- 100% запланованих тест-кейсів для поточної фази виконано.
- Відсутні дефекти рівня `Blocker` і `Critical`.
- Не менше 95% функціональних тестів завершилися статусом `Passed`.
- Всі дефекти рівня `Major`, що впливають на `login`, `punch-in`, `punch-out`, `time corrections`, `dashboard`, `csv export`, виправлені або погоджені із задокументованим обхідним шляхом.
- Нефункціональні цілі виконані: базові API-відповіді для ключових операцій стабільно укладаються в межу до 2 секунд у локальному тестовому середовищі без штучного навантаження.

Фаза вважається неуспішною, якщо:

- виявлено хоча б один дефект, що унеможливлює запуск системи або базовий бізнес-потік;
- тестова збірка не стартує, не підключається до БД або повертає системні помилки на ключових маршрутах;
- частка невдалих тестів перевищує 5% для затвердженого набору;
- результати звітності, тривалості зміни або авторизації не відповідають вимогам SRS чи фактичній бізнес-логіці.

## 7. Entry and Exit Criteria

### Entry
- Кодова база зібрана та проходить синтаксичну перевірку без помилок.
- Сервер стартує успішно через `node server.js`.
- Тестова БД `WebTimeTrackerDB` доступна за `mongodb://localhost:27017` або за узгодженим URI.
- Підготовлено тестові дані щонайменше для 2 компаній, 3 користувачів, активних та завершених змін, а також запитів на коригування.
- Узгоджено тестові сценарії, критерії приймання та ролі учасників.
- Налаштовано інструменти ведення дефектів і фіксації результатів тестування.

**Примітка:** станом на 2026-04-07 умова входу ще не виконується повністю, оскільки в `app.js` присутній синтаксичний дефект: сторонній текстовий фрагмент після маршруту `POST /api/time-logs/punch-out`, що блокує перевірку `node -c app.js`.

### Exit
- Усі обов’язкові тест-кейси виконані та протоколізовані.
- Дефекти `Blocker` і `Critical` закриті, а залишкові `Major/Minor` погоджені.
- Перевірено ключові бізнес-потоки: авторизація, панчі, коригування, дашборд, CSV-звіт.
- Сформовано підсумковий звіт тестування та рекомендацію щодо релізу.
- Дані в БД після тестування очищено або архівовано, якщо це передбачено регламентом середовища.

## 8. Test Environment

Тестове середовище повинно відповідати такій конфігурації:

- **ОС тестового стенду:** `Windows 11 x64` для локального QA-виконання; додатково рекомендовано віддзеркалене середовище `Ubuntu 22.04 LTS` для передрелізної перевірки серверного розгортання.
- **Серверна платформа:** `Node.js v24.14.0`, запуск через `node server.js`, модульний формат `CommonJS`.
- **Backend framework:** `Express 5.2.1`.
- **База даних:** `MongoDB`, база `WebTimeTrackerDB`, локальне підключення `mongodb://localhost:27017`.
- **Колекції БД:** `companies`, `users`, `time_logs`, `time_corrections`.
- **Стан індексації:** фактично наявні лише стандартні індекси `_id`; прикладні індекси на `email`, `company_id`, `user_id`, `date`, `organization_code` відсутні.
- **Клієнтські браузери для приймального тестування:** останні стабільні версії `Google Chrome`, `Mozilla Firefox`, `Microsoft Edge`; для кросплатформної перевірки бізнес-вимог також `Safari` на macOS/iOS.
- **Мінімальні серверні ресурси:** `2 vCPU`, `4 GB RAM`, `50 GB SSD`, відповідно до вимог із лабораторної роботи №2.
- **Мережеві умови:** стабільне локальне або корпоративне з’єднання; для базового перформанс-тесту допустима затримка до 50 мс у межах локальної мережі без втрати пакетів.
- **Інструменти тестування:** `Postman` або `Bruno`, `Newman`, `MongoDB Compass`, `curl`, `PowerShell`, `Jest` або `node:test` + `Supertest`, `k6`, `OWASP ZAP`, засіб трекінгу дефектів (`Jira`, `GitHub Issues` або аналог).

## 11. Risks and Mitigations

| Ризик | Вплив | Ймовірність | Стратегія пом’якшення |
| --- | --- | --- | --- |
| Синтаксичний дефект у `app.js` блокує запуск сервера | Високий | Висока | Усунути дефект до старту системного тестування; додати перевірку `node -c` у pre-commit або CI. |
| Відсутність автоматизованих тестів у репозиторії | Високий | Висока | Створити мінімальний набір API smoke/regression tests для ключових маршрутів. |
| Тестові дані в MongoDB можуть містити placeholder-хеші або неповні поля | Високий | Середня | Підготувати контрольований seed-набір із валідними bcrypt-хешами, `organization_code` і реалістичними ролями. |
| Відсутність прикладних індексів у MongoDB погіршує швидкодію при масштабуванні | Середній | Висока | Додати індекси на `users.email`, `companies.organization_code`, `time_logs.company_id`, `time_logs.user_id`, `time_logs.date`, `time_corrections.status`. |
| Ризик некоректної роботи з датами, часовими поясами та `date-only` полями | Високий | Середня | Провести окремі тести на `UTC`, `Europe/Kyiv`, переходи дат, граничні значення та нічні зміни. |
| Відсутність рольової авторизації на частині адміністративних маршрутів | Високий | Середня | Додати тести безпеки та зафіксувати як дефект/борг до релізу. |
| Автоматичне прив’язування коригування до логу може давати конфлікти при кількох змінах за день | Середній | Середня | Передбачити негативні тести на `409` і вимагати явний `time_log_id` у неоднозначних випадках. |
| Відсутність виділеного staging-середовища | Середній | Середня | Розгорнути окрему БД та конфігурацію для QA/UAT, не використовувати спільні робочі дані. |

## Висновок

Запропонований план тестування охоплює поточний стан проєкту `Web Time Tracker (Puncher API)` як backend-ядра системи автоматизованого обліку робочого часу. Документ узгоджує бізнес-вимоги попередніх лабораторних робіт із фактично реалізованими маршрутами, MongoDB-схемою та реально виявленими технічними ризиками. За умови усунення поточного синтаксичного дефекту, підготовки валідних тестових даних і впровадження базового автотестового контуру цей план може бути використаний як офіційна основа для лабораторного репозиторію та подальшого приймального тестування.
